BUG 2 — Operator precedence error in __lt__ (somatic value sort is broken)
FILE:    classification/models/evidence_mixin.py
LINE:    67
IMPACT:  HIGH — somatic classification values sort incorrectly
--------------------------------------------------------------------------------

    def __lt__(self, other):
        return self.sort_value or 0 < other.sort_value or 0

Python operator precedence makes this parse as:
    return (self.sort_value) or (0 < other.sort_value) or (0)

Which is a boolean/truthy evaluation, NOT a numeric comparison. The intended
logic is clearly:
    return (self.sort_value or 0) < (other.sort_value or 0)

Fix: add parentheses around each "or 0" clause.

SECTION 2: ARCHITECTURAL SUGGESTIONS (high benefit / low cost)
================================================================================

These are suggestions for improvements where the payoff is large relative to
the implementation effort. They are listed roughly in order of benefit/cost ratio.

--------------------------------------------------------------------------------
SUGGESTION 1 — Eliminate the parallel clinical significance value systems
BENEFIT: HIGH  |  COST: LOW-MEDIUM
--------------------------------------------------------------------------------

There are currently two separate representations of clinical significance:

  a) ClinicalSignificance class (classification_enums.py):
       BENIGN = '1', LIKELY_BENIGN = '2', VUS = '3', LP = '4', P = '5'

  b) EvidenceKey option keys used in the evidence JSON blob:
       'B', 'LB', 'VUS', 'LP', 'P'  (plus 'VUS_A', 'VUS_B', etc.)

These two systems coexist and are used in different code paths, leading to
subtle confusion (e.g. one of the agent-reported false positives mistook
the 'VUS' string check in is_significant_change() for a bug because it
looked like it should be ClinicalSignificance.VUS == '3').

The ClinicalSignificance class (system a) appears to be a legacy/DB-level
concept while system b is the live representation. Clearly marking which
contexts each applies in, or consolidating them, would reduce cognitive load
and the risk of real bugs at the boundary.

Minimum viable improvement: add a docstring to ClinicalSignificance explaining
when to use it vs EvidenceKey values, and add a conversion helper.

SUGGESTION 3 — Protect EvidenceKeyMap cache invalidation in multi-worker deploys
BENEFIT: HIGH  |  COST: LOW
--------------------------------------------------------------------------------

EvidenceKeyMap uses @cached_property / class-level caching. In production,
multiple Gunicorn workers each hold their own in-process cache. If an admin
updates EvidenceKey records, the change propagates on next request to some
workers but not others until they restart.

Options (cheapest first):
  a) Add a management command to touch a cache-bust key in Redis, and document
     that EvidenceKey changes require a worker restart or cache clear.
  b) Move the cache to Redis (django.core.cache) keyed on a version counter
     stored in the DB, so all workers see the same version.


--------------------------------------------------------------------------------
SUGGESTION 4 — Move view-layer helper dataclasses out of discordance_report_views.py
BENEFIT: MEDIUM  |  COST: LOW
--------------------------------------------------------------------------------

discordance_report_views.py contains substantial non-view logic:
  - _LabClinSigKey, _LabClinSig (dataclasses for lab/clin-sig grouping)
  - DiscordanceReportTemplateData (200+ line class that aggregates report data)

This makes the view file hard to navigate and the logic hard to test in
isolation. Moving DiscordanceReportTemplateData to
classification/models/discordance_models_utils.py (which already exists)
and the helper dataclasses to a classification/views/discordance_helpers.py
would improve separation and testability.


--------------------------------------------------------------------------------
SUGGESTION 5 — Make classification import task chain explicit with Celery chords
BENEFIT: MEDIUM  |  COST: MEDIUM
--------------------------------------------------------------------------------

The import pipeline is spread across three separate Celery tasks:
  1. ClassificationImportMapInsertTask   (map file → VG JSON, then insert)
  2. process_classification_import_task  (process one ClassificationImport)
  3. ClassificationImportProcessVariantsTask  (link variants, trigger liftover)

The dependencies between them are implicit (each task creates DB state the
next reads). Expressing this as a Celery chain/chord would:
  - Make the dependency graph explicit in code
  - Enable automatic retry/error propagation across the whole pipeline
  - Make it easier to observe progress of a full import run


--------------------------------------------------------------------------------
SUGGESTION 6 — Move discordance recalculation to a Celery task
BENEFIT: MEDIUM  |  COST: LOW-MEDIUM
--------------------------------------------------------------------------------

ClinicalContext.recalculate() is called synchronously on every classification
publication (via signal). For labs with hundreds of classifications for the
same allele, this blocks the request/response cycle. Moving this to an async
Celery task (with a short delay to batch rapid publications) would improve
response times for bulk imports and the import API.

The ClinicalContextRecalcTrigger model already exists and seems designed to
support deferred recalculation — it may be partially implemented.


--------------------------------------------------------------------------------
SUGGESTION 7 — Consolidate the two allele_origin filtering paths in grouping datatables
BENEFIT: LOW-MEDIUM  |  COST: LOW
--------------------------------------------------------------------------------

In classification_grouping_datatables.py, there are two separate code paths
for filtering groupings by variant:
  - gene_symbol filter (line ~205+): traverses allele_grouping correctly
  - protein_position filter (line 203): traverses allele_grouing (typo, Bug 4)

Once Bug 4 is fixed, both paths use the same relation chain. Consider
extracting the shared allele→variant→annotation join into a helper method
to avoid future divergence.


--------------------------------------------------------------------------------
SUGGESTION 8 — Typed evidence blob accessors
BENEFIT: MEDIUM  |  COST: MEDIUM
--------------------------------------------------------------------------------

The evidence JSONField stores all classification data as an untyped dict.
Accessing values (get/set via EvidenceMixin) works but provides no IDE
autocompletion or type safety for individual keys.

A low-effort improvement: create TypedDict definitions for the most-used
special keys (e.g. ClinSigBlob, CHGVSBlob) and use them in the hot paths
(classification_hooks_significant_change, classification_grouping, etc.).
This does not require schema migration and can be done incrementally.


--------------------------------------------------------------------------------
SUGGESTION 9 — Rename confusing loop variables in string_to_labs / string_to_orgs
BENEFIT: LOW  |  COST: VERY LOW
--------------------------------------------------------------------------------

In classification_export_view.py lines 208–219, the list comprehension loop
variable is named lab_str, shadowing the function parameter of the same name:

    def string_to_labs(lab_str: str) -> list[Lab]:
        parts = [l.strip() for l in lab_str.split(',')]
        labs = [Lab.objects.filter(group_name=lab_str).first() for lab_str in parts]

Python 3 comprehension scoping means the filter correctly uses the loop
variable (not the parameter), so this is NOT a functional bug. But the
naming is actively misleading to readers. Renaming the loop variable to
`part` in both functions would make the intent clear.
